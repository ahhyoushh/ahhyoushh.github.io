Arrays & Hashing
9 problems
Hash maps for O(1) lookup. Hash sets for duplicate detection. The naive approach teaches you when NOT to use a hash.
Key C++ concepts: unordered_map, unordered_set, reserve(), max_load_factor(). Hash tables have O(1) amortized but O(n) worst case. Sorting gives O(n log n) but O(1) space — always weigh the tradeoff. 
// Hash set approach — O(n) time, O(n) space
unordered_set<int> seen;
seen.reserve(nums.size());  // avoid rehashing
for (int n : nums)
    if (!seen.insert(n).second) return true;

// Sort approach — O(n log n) time, O(1) space
sort(nums.begin(), nums.end());
for (int i = 0; i < (int)nums.size() - 1; i++)
    if (nums[i] == nums[i+1]) return true;
Two Pointers
5 problems
Two indices moving toward each other or in the same direction. Eliminates O(n) space by working in-place on sorted data.
Key insight: Sorting first costs O(n log n) but unlocks O(1) space solutions. The CPU cache loves sequential access — two pointers on a sorted array is cache-optimal. unordered_set is cache-hostile (hash jumps). 
// 3Sum — O(n²) time, O(1) extra space
sort(nums.begin(), nums.end());
for (int i = 0; i < n - 2; i++) {
    int lo = i+1, hi = n-1;
    while (lo < hi) {
        int s = nums[i] + nums[lo] + nums[hi];
        if (s == 0) { /* found */ }
        else if (s < 0) lo++;
        else hi--;
    }
}
Sliding Window
6 problems
Maintain a variable-size window over a sequence. Expand right, shrink left based on constraint. O(n) from O(n²).
Key insight: Each element enters and exits the window at most once → O(n). The window lives in registers/stack. The array is read sequentially → perfect cache behavior. This is why it beats brute force not just in big-O but in real wall time. 
// Longest substring without repeating chars
int l = 0, res = 0;
unordered_map<char,int> seen;
for (int r = 0; r < s.size(); r++) {
    if (seen.count(s[r])) l = max(l, seen[s[r]]+1);
    seen[s[r]] = r;
    res = max(res, r - l + 1);
}
Stack
7 problems
LIFO structure for tracking "pending" decisions. Monotonic stack is the advanced form — maintains sorted order as you push.
Architecture note: std::stack wraps std::deque by default (heap allocation, cache-unfriendly). Prefer stack<int, vector<int>> for better cache behavior. The call stack itself is a stack — recursive DFS uses the CPU's native stack which is fast. 
// Monotonic decreasing stack — Daily Temperatures
stack<int, vector<int>> st;  // vector backend = cache friendly
for (int i = 0; i < n; i++) {
    while (!st.empty() && temps[i] > temps[st.top()]) {
        res[st.top()] = i - st.top();
        st.pop();
    }
    st.push(i);
}
Binary Search
7 problems
Not just sorted arrays — it's a framework for searching any monotonic answer space. "Is X feasible?" → binary search on X.
The overflow trap: Every C++ binary search in an interview needs the safe midpoint. Also: std::lower_bound and std::upper_bound are O(log n) and already correct — know when to use them vs manual. 
// ❌ OVERFLOW — classic mistake
int mid = (lo + hi) / 2;  // UB if lo+hi > INT_MAX

// ✅ SAFE
int mid = lo + (hi - lo) / 2;

// Binary search on answer space (Koko Eating Bananas)
int lo = 1, hi = *max_element(piles.begin(), piles.end());
while (lo < hi) {
    int mid = lo + (hi - lo) / 2;
    if (feasible(piles, mid, h)) hi = mid;
    else lo = mid + 1;
}
Linked Lists
11 problems
Pointer manipulation. Fast/slow pointers. Reverse in-place. The most memory-explicit pattern — teaches you heap allocation directly.
Architecture note: Linked list nodes are heap-allocated, scattered in memory → cache misses on every node access. This is why std::vector dominates in practice. But understanding linked list manipulation is fundamental to understanding heap layout and pointer safety. 
// Reverse in-place — O(n) time, O(1) space
ListNode* reverseList(ListNode* head) {
    ListNode* prev = nullptr;
    while (head) {
        auto* nxt = head->next;
        head->next = prev;
        prev = head;
        head = nxt;
    }
    return prev;
}

Write it. Submit it. Get the green checkmark. This is your baseline. Do not optimize yet.
4
0:20 – 0:27
Optimize — Eliminate the Allocation
Can I eliminate the hash set? (Two pointers / sort.) Can I reduce from O(n²) to O(n log n) or O(n)? Am I doing any unnecessary copies? Rewrite, get accepted.
5
0:27 – 0:30
Final Check — The Four Questions
(1) Are all input params passed as const& where not modified?
(2) Any overflow risk? (n > 10⁴ with multiplication → use long long)
(3) Am I using the right STL container for the access pattern?
(4) Can I explain the memory layout of my solution?

C++ Habits & Anti-Patterns
Every mistake here is a real bug in a codebase somewhere.
Anti-Pattern What to Do Instead Why It Matters
vector<T> by value in param const vector<T>& Passing by value copies the entire vector on every call. O(n) hidden cost.
(lo + hi) / 2 lo + (hi - lo) / 2 lo + hi overflows INT_MAX when both are large. Undefined behavior.
int with large multiplication long long when n > 10⁴ 10⁵ × 10⁵ = 10¹⁰ which overflows 32-bit int silently.
cout << endl cout << "\n" endl flushes the buffer every call. "\n" does not. Significant in tight loops.
for (int i=0; i < v.size(); i++) for (int i=0; i < (int)v.size(); i++) size() returns size_t (unsigned). Comparing int to unsigned causes subtle bugs when i goes negative.
unordered_set for everything Sort + two pointers when space matters unordered_set has O(1) avg but ~149 MB overhead on large inputs. Sort is in-place.
stack<int> (default deque backend) stack<int, vector<int>> deque is heap-fragmented. vector backend is contiguous → cache friendly.
Not reserving unordered containers seen.reserve(nums.size()) Without reserve, multiple rehashes occur. Reserve eliminates them entirely.
